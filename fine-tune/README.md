# jAIcianVerse — Llama 3.2 LoRA Fine-Tuning Pipeline

End-to-end pipeline to fine-tune **Meta Llama 3.2 3B Instruct** on a domain-specific college knowledge base so the model **internalises** the information (no RAG required at inference time).

```
TXT  ──▶  Dataset (JSONL)  ──▶  LoRA Fine-Tune  ──▶  GGUF  ──▶  Ollama Model
```

## 1. Model Selection & Architecture

### Base Model

| Field | Value |
|---|---|
| HuggingFace ID | `unsloth/Llama-3.2-3B-unsloth-bnb-4bit` |
| Architecture | `LlamaForCausalLM` |
| Parameter count | **3 billion** |
| Quantisation (training) | **4-bit NF4** via `bitsandbytes` (BnB) |

### Model Architecture (from `gguf_output/merged_hf/config.json`)

| Parameter | Value |
|---|---|
| `hidden_size` | 3072 |
| `intermediate_size` | 8192 |
| `num_hidden_layers` | 28 |
| `num_attention_heads` | 24 |
| `num_key_value_heads` | 8 (Grouped Query Attention) |
| `head_dim` | 128 |
| `hidden_act` | `silu` |
| `vocab_size` | 128 256 |
| `rms_norm_eps` | 1e-5 |
| `max_position_embeddings` | 131 072 |
| `rope_theta` | 500 000.0 |
| RoPE scaling type | `llama3` (factor 32.0, original max 8192) |
| `tie_word_embeddings` | true |
| `torch_dtype` | float16 |
| BOS token id | 128 000 |
| EOS token id | 128 001 |
| PAD token id | 128 004 |

### Why Llama 3.2 3B?

- Fits comfortably on a single consumer GPU at 4-bit (< 4 GB VRAM).
- Llama 3.2 uses the same tokeniser and chat template as Llama 3.1/3.0, so the existing Modelfile template works without changes.
- Unsloth provides a pre-quantised BnB 4-bit build that reduces loading time and memory overhead.
- The 3B model is fast enough for a college chatbot use case without sacrificing too much answer quality.

## 3. File-by-File Reference

### 3.1 `prepare_dataset.py`

**Purpose:** Reads a plain-text knowledge file, splits it into overlapping chunks, and generates instruction-tuning Q&A pairs using one of three backends.

#### How It Works

1. **Text chunking** — `split_into_chunks(text, max_tokens=512, overlap_sentences=1)` splits by paragraphs and sentences. Each chunk stays under the token limit, and one sentence of overlap is added between consecutive chunks to prevent context fragmentation.
2. **Q&A generation** — for each chunk a prompt is built asking the LLM to produce 2–5 Q&A pairs as JSON (`_build_qa_prompt`).
3. **Output** — each pair is written to JSONL:
   ```json
   {"instruction": "question text", "input": "", "output": "answer text"}
   ```

#### Three Backends

| Backend | Function | Details |
|---|---|---|
| `ollama` (default) | `generate_qa_with_ollama` | Calls local Ollama API at `http://localhost:11434/api/generate`; model configurable (`--ollama-model`); `temperature=0.3`, `num_predict=1024` |
| `gemini` | `generate_qa_with_gemini` | Calls Google Gemini API; default model `gemini-2.5-flash-lite`; `temperature=0.3`, `max_output_tokens=1024` |
| `rulebased` | `generate_qa_rulebased` | Heuristic fallback — generates "What do you know about…?" style pairs; no LLM needed |

#### Key Imports

```
argparse, json, os, re, sys, time, textwrap
pathlib.Path, typing.Optional
dotenv.load_dotenv
requests                  # for Ollama backend
google.generativeai       # for Gemini backend
```

#### CLI Usage

```bash
# Ollama backend (default)
python prepare_dataset.py \
  --input ../data/college.data.txt \
  --output dataset.jsonl \
  --backend ollama \
  --ollama-model llama3

# Gemini backend
python prepare_dataset.py \
  --input ../data/college.data.txt \
  --output dataset.jsonl \
  --backend gemini \
  --gemini-api-key YOUR_KEY

# Rule-based (no LLM)
python prepare_dataset.py \
  --input ../data/college.data.txt \
  --output dataset.jsonl \
  --backend rulebased
```

### 3.2 `train_lora.py`

**Purpose:** Fine-tunes the base model with LoRA adapters using Unsloth (optimised 4-bit training) and TRL's `SFTTrainer`.

#### How It Works

1. **Hardware detection** — `detect_device()` checks for CUDA availability and selects `per_device_train_batch_size=2` on GPU or `1` on CPU; sets `bf16`/`fp16` flags automatically.
2. **Model loading** — `FastLanguageModel.from_pretrained(...)` loads the base model in 4-bit NF4 with Unsloth's CUDA kernels (~3–4 GB VRAM).
3. **LoRA injection** — `FastLanguageModel.get_peft_model(...)` injects Low-Rank Adapters into all 7 attention and MLP projection layers.
4. **Dataset formatting** — `format_prompts(examples)` applies the full **Llama 3 Instruct chat template**:
   ```
   <|begin_of_text|><|start_header_id|>system<|end_header_id|>
   You are a helpful assistant.<|eot_id|>
   <|start_header_id|>user<|end_header_id|>
   {instruction}<|eot_id|>
   <|start_header_id|>assistant<|end_header_id|>
   {output}<|eot_id|>
   ```
5. **Training loop** — `SFTTrainer` + `SFTConfig` from TRL handles the supervised fine-tuning loop with gradient checkpointing (`"unsloth"` mode), 8-bit AdamW, and cosine LR scheduling.
6. **Saving** — the LoRA adapter (not the full model) is saved to `./lora_output/lora_adapter/` (≈93 MB). Full checkpoints are saved at every 100 steps (max 2 kept).

#### Windows-Specific Workarounds

The script redirects Triton and TorchInductor cache directories to `%LOCALAPPDATA%\triton_cache` to avoid Windows path-length issues.

#### Key Imports

```
argparse, json, os, sys, pathlib.Path
dotenv.load_dotenv
torch
unsloth.FastLanguageModel
datasets.Dataset
trl.SFTTrainer, trl.SFTConfig
```

#### CLI Usage

```bash
python train_lora.py \
  --dataset dataset.jsonl \
  --output-dir ./lora_output \
  --base-model unsloth/Llama-3.2-3B-unsloth-bnb-4bit \
  --lora-rank 16 \
  --lora-alpha 32 \
  --learning-rate 2e-4 \
  --epochs 3
```

### 3.3 `export_model.py`

**Purpose:** Merges the trained LoRA adapter back into the base model weights and exports the result as a single GGUF file usable by Ollama.

#### How It Works

1. **Load + merge** — `merge_and_save_hf(adapter_dir, gguf_dir)` reloads the base model + adapter via Unsloth in 4-bit, then calls `model.save_pretrained_merged(..., save_method="merged_16bit")` to produce a full 16-bit HuggingFace checkpoint in `gguf_output/merged_hf/`.
2. **GPU flush** — `_free_gpu()` calls `torch.cuda.empty_cache()` and `gc.collect()` between the two heavy steps.
3. **GGUF conversion** — `convert_hf_to_gguf(merged_dir, gguf_dir)` locates `convert_hf_to_gguf.py` (from llama.cpp, found via `_find_convert_script()`) and calls it via subprocess with `--outtype f16` producing `gguf_output/unsloth.f16.gguf`.
4. **Fallback** — if the llama.cpp script is not on PATH it searches `~/.unsloth/llama.cpp/`.

> **Note:** No C++ compiler is required on Windows. Ollama handles quantisation at model-import time (`ollama create --quantize q4_K_M`).

#### Key Imports

```
argparse, gc, shutil, subprocess, sys, pathlib.Path
torch        (lazy-imported)
unsloth.FastLanguageModel  (lazy-imported)
```

#### CLI Usage

```bash
python export_model.py \
  --adapter-dir ./lora_output/lora_adapter \
  --gguf-dir ./gguf_output \
  --max-seq-length 2048
```

## 8. GGUF Export & Inference Config

### Merge Process

`export_model.py` merges the 93 MB adapter into the 3B base weights producing a single full 16-bit model:

| Artifact | Size |
|---|---|
| `gguf_output/merged_hf/` (HF shards) | ~6.43 GB (2 shards) |
| `gguf_output/unsloth.f16.gguf` | **6.0 GB** |

### Quantisation

The GGUF is exported as **F16** (full precision floating-point 16). Ollama applies further quantisation at `ollama create` time:

```bash
ollama create jaicianverse -f Modelfile --quantize q4_K_M
```

Recommended quantisation levels for Ollama:

| Level | Size (approx.) | Quality |
|---|---|---|
| `q4_k_m` | ~2.0 GB | **Recommended** — best speed/quality trade-off |
| `q5_k_m` | ~2.4 GB | Higher quality, slightly slower |
| `q8_0` | ~3.5 GB | Near lossless, requires more VRAM |
| `f16` | ~6.0 GB | Original export; full precision |

### Ollama Model Name

```
jaicianverse
```

## 14. Troubleshooting

| Issue | Fix |
|---|---|
| `OutOfMemoryError` during training | Reduce `--max-seq-length` to 1024 or use `--lora-rank 8` |
| Unsloth import error | Ensure `bitsandbytes` and `xformers` match your CUDA version exactly |
| `GGUF export fails` | Verify `convert_hf_to_gguf.py` from llama.cpp is on PATH or in `~/.unsloth/llama.cpp/` |
| Ollama gives generic answers | Increase epochs to 5, or regenerate dataset with `--backend gemini` for higher quality pairs |
| Dataset has low-quality pairs | Switch from `rulebased` to `ollama` or `gemini` backend |
| Triton / cache errors on Windows | Script auto-redirects cache to `%LOCALAPPDATA%\triton_cache`; ensure that path is writable |
| `adapter_model.safetensors` not found | Run `train_lora.py` first; do not point `--adapter-dir` at a checkpoint folder, use `lora_adapter/` |
| Modelfile FROM path mismatch | Edit the `FROM` line in `Modelfile` to match the actual GGUF filename in `gguf_output/` |
